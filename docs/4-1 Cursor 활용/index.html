2. 주요 기능

  1. Agent 모드 (Cmd + I - Composer)

특징

  • 다중 파일 동시 편집 가능
  • 자율적인 코드 생성 및 수정
  • 프로젝트 구조 이해 기반 작업

사용법

  • Cmd + I 로 Composer 열기
  • 원하는 작업을 자연어로 설명
  • AI가 자동으로 여러 파일을 분석하고 수정
  •  변경사항 검토 후 Apply 또는 Rejec


실사용 예시

"노트앱에 다크모드 기능을 추가해줘."

  • 테마 전환 버튼 추가 - CSS 변수로 색상 관리 - 사용자 설정 저장 - 모든 컴포넌트에 적용
  • AI가 자동으로 여러 파일을 수정: - theme.css 생성
  • components/ThemeToggle.jsx 생성
  • App.jsx 수정 - localStorage 로직 추가
  • 토큰 사용량: 높음 (5000-15000 토큰)
  • 전체 프로젝트 컨텍스트 분석
  • 다중 파일 동시 처리
  • 복잡한 로직 생성


2. Ask 모드 (Cmd + L - Chat)

특징

  • 질문-답변 중심의 대화형 인터페이스
  • 코드 설명, 디버깅, 리뷰 전문
  • 참조 파일 자동 선택

사용법

  • Cmd + L로 채팅창 열기
  • @ 기호로 특정 파일 참조
  • 질문이나 요청 입력
  • AI가 답변과 함께 코드 예시 제공


실사용 예시

"@src/components/NoteEditor.jsx 이 컴포넌트에서 메모리 누수가 발생할 수 있는 부분이 있을까?"

→ AI 응답: "네, 몇 가지 잠재적 메모리 누수 지점이 있습니다:

  • 1. useEffect에서 이벤트 리스너 정리 안 됨
  • 2. setTimeout 정리 안 됨
  • 3. WebSocket 연결 정리 안 됨
  • 4. [구체적인 수정 코드 제시]"
  • 토큰 사용량: 중간 (1000-5000 토큰)
  • 참조된 파일들만 컨텍스트로 사용
  • 질문에 따라 사용량 차이



3. Manual 모드 (Cmd + K - Quick Edit)


특징

  • 선택된 코드만 집중 편집
  • 빠른 수정 및 개선
  • 최소한의 컨텍스트 사용

사용법

  • 수정하고 싶은 코드 선택
  • Cmd + K 입력
  • 구체적인 수정 요청 입력
  • AI가 해당 부분만 수정
// 선택된 코드 - "이 코드에 에러 처리를 추가해줘"
function saveNote(note) {
    localStorage.setItem('note', JSON.stringify(note));
}

// AI 결과
function saveNote(note) {
    try {
        if (!note || typeof note !== 'object') {
            throw new Error('Invalid note object');
        }
        localStorage.setItem('note', JSON.stringify(note));
        return true;
    } catch (error) {
        console.error('Failed to save note:', error);
        return false;
    }
}


3. 개발 및 디버깅

문제가 되는 부분의 코드나 고치고 싶은 부분이 코드를 개발자 도구 (F12)를 열어 Element Inspect 를 통해 영역을 지정해서 수정하거나 기능을 추가할 수 있다.


해당 엘리먼트를 직접 찾아서 Add to Chat을 통해 해당 라인을 채팅 모드에 추가할 수 있으며 Quick Edit로도 수정이 가능하다.

여기서는 계산기 웹 화면에서 연산자 입력시 화면에 표시가 안되고 계산을 하는 불편함이 있으므로 연산자도 클릭시 화면에 노출하도록 해당 코드에 수정할 스펙 프롬프트를 입력해주고 수정을 진행해보도록 한다.


만약 코드 변경이 지지부진하거나 제대로 안되는 느낌이 들면 에이전트 모델을 상위 모델로 변경해야 한다.


빠른 속도로 작업하길 원한다면 Auto 모드를 사용해서 간단한 버그들은 처리할 수 있다. 


현재 커서에서 제일 상위 모델은 claude-4.5-sonnet 모델이고, MAX 모드로 선택한 경우 더 원할한 기능을 수행할 수 있으나 Token의 소모가 심하다는 단점이 있다.

따라서 일반적인 작업이라면 Auto 모드로 해도 충분히 수정이 가능할 것이고, 속도 또한 만족할만큼 빠르므로 참고해서 사용해보도록 한다.

또, 가끔 에이전트가 엉뚱한 작업을 하거나 전혀 의도하지 않은 작업들을 수행한다고 느껴질때가 있는데 이럴 경우 대화 내역을 다 리셋하고, 코드 역시 롤백하고 초기부터 다시 작업하는게 유리하다.

5. Cursor Rules 세팅

대규모 프로젝트가 아니더라도 반드시 개발 환경에 대한 rule을 세팅하는 것을 추천한다.

룰은 커서 세팅 화면에서 Rules & Memories 에서 설정할수 있고 프로젝트 루트 폴더 하위에 .cursor/rules에 작성해서 적용할 수 있다.

각 기술셋에 대한 기본적인 Rule 세팅은 다음의 링크를 참고하기 바란다.

https://github.com/PatrickJS/awesome-cursorrules?tab=readme-ov-file


프로젝트 규칙을 사용하여 다음을 수행할 수 있다.

  • 코드베이스에 대한 도메인별 지식을 인코딩
  • 프로젝트별 워크플로 또는 템플릿 자동화
  • 스타일 또는 아키텍처 결정 표준화

.MDC 파일로 작성되어야 하며 다른 룰을 참고에 포함하여 트리거 할 수 있다.

---
description: RPC Service boilerplate
globs:
alwaysApply: false
---

- Use our internal RPC pattern when defining services
- Always use snake_case for service names.

@service-template.ts

service-template.ts 규칙을 포함하여 추가로 컨텍스트가 유지된다.

또한 프로젝트 전체 티렉토리에 중첩해서 규칙을 적용할 수 있다.

project-root/
  .cursor/rules/        # Project-wide rules
  backend/
    server/
      .cursor/rules/    # Backend-specific rules
  frontend/
    .cursor/rules/      # Frontend-specific rules

Setting에 추가하기 번거롭다면 검색을 통해 다양한 Rule 파일들을 복사해서 적용해도 무방하고, Cursor 안에서 /Generate Cursor Rules 명령어를 통해서 적용해도 상관없다.



다음은 Rule 샘플 파일이다.

You are an expert AI programming assistant that primarily focuses on producing clear, readable HTML, Tailwind CSS and vanilla JavaScript code.

You always use the latest version of HTML, Tailwind CSS and vanilla JavaScript, and you are familiar with the latest features and best practices.

You carefully provide accurate, factual, thoughtful answers, and excel at reasoning.

- Follow the user's requirements carefully & to the letter.
- Confirm, then write code!
- Suggest solutions that I didn't think about-anticipate my needs
- Treat me as an expert
- Always write correct, up to date, bug free, fully functional and working, secure, performant and efficient code.
- Focus on readability over being performant.
- Fully implement all requested functionality.
- Leave NO todo's, placeholders or missing pieces.
- Be concise. Minimize any other prose.
- Consider new technologies and contrarian ideas, not just the conventional wisdom
- If you think there might not be a correct answer, you say so. If you do not know the answer, say so instead of guessing.
- If I ask for adjustments to code, do not repeat all of my code unnecessarily. Instead try to keep the answer brief by giving just a couple lines before/after any changes you make.
- All responses will be in Korean.

General Component Creation Guidelines:

General Rules:

- Enable strict TypeScript (strict: true in tsconfig.json)
- Avoid 'any', prefer 'unknown' with runtime checks
- Explicitly type function inputs and outputs
- Use advanced TypeScript features (type guards, mapped types, conditional types)

Next.js Rules:

6. Best Practice 와 예시

좋은 규칙은 집중적이고, 실행 가능하며, 범위가 명확해야 한다. 아래와 같은 명확한 룰을 작성해야 한다.

  • 규칙을 500줄 이하로 유지하세요
  • 큰 규칙을 여러 개의 구성 가능한 규칙으로 분할
  • 구체적인 예나 참조 파일을 제공하세요
  • 모호한 지침은 피하세요. 명확한 내부 문서와 같은 규칙을 작성하세요.
  • 채팅에서 프롬프트를 반복할 때 규칙을 재사용하세요.



Tip.

일반적인 LLM 에도 역할을 부여하고 설명상의 규칙을 넣듯이 개발하고자 하는 플랫폼과 언어, 프로젝트에 맞도록 룰을 부여하는게 코드생성에서 훨씬 좋은 결과를 낼 수 있다.

가급적 프로젝트를 열고 작업할 때 Cursor Settings → Indexing & Docs 메뉴를 통해 코드베이스를 인덱싱하고 rule 파일을 다시 한번 Cursor에 인식시키는 습관을 들이도록 한다.

중요 정보를 다루는 파일이 있다면 .cursorignore 파일에 추가해서 엑세스를 제어할 수 있다. (.cursorindexignore, .gitignore 등 비슷한 역할을 한다)

가급적 유명한 Repository와 Boilerplate 를 참고해서 시작하는것이 좋다.
