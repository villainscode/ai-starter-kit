기본 개발 방법 가이드 

Markdown 혹은 Markdown Components 를 이용하여 지침을 제공하고, AI Agent로 하여금 개발 가이드를 지켜 일관성 있는 제품을 구현하는데 집중한다.


파일 형식 차이 

.md (Markdown)

  • 표준 마크다운 파일
  • 순수 텍스트 기반 포맷
  • 어디서나 사용 가능한 범용 형식
  • HTML로 변환이 가능하지만 기능 자체는 제한적
  • GitHub, Notion, VSCode 등 모든 플랫폼 지원

.mdc (Markdown Components)

  • 특정 프레임 워크에서 좀 더 복잡한 표현을 통해 동작하고자 할 때 
  • Vue나 React 등의  컴포넌트를 마크다운에 포함 가능
  • 메타데이터 확장 지원
  • 주로 웹 문서화 시스템에서 사용


MDC의 경우 yaml 형태로 메터데이터를 구조화 할 수 있어 AI Agent가 문서의 계층 관계를 파악하기 쉽다는 장점이 있으나 주로 md로 제공해도 문제가 없다 (더 간편)


필수 문서 파일 

- AGENTS.md (모든 Agent)

- CLAUDE.md (Claude code 의 경우) 

- .cursor > roles > rules.mdc (Cursor) 

각 문서들은 챕터 4의 각 Agent에서 상세한 예시를 따로 정리함

# AGENTS.md (Java Backend Version)
## 📂 프로젝트 구조 & 모듈 구성

-   `src/main/java/`: Spring Boot 애플리케이션 메인 코드 (Controller,
    Service, Repository, Domain)
-   `src/main/resources/`: 설정 파일 (`application.yml`, `schema.sql`,
    `data.sql`)
-   `src/test/java/`: 단위 테스트 및 통합 테스트 (JUnit + Mockito)
-   `config/`: 공통 환경 설정 (DB, Cache, MQ, Security)
-   `docs/`: 아키텍처, API 스펙, ERD, 시퀀스 다이어그램 등 문서
-   `build/`, `out/`, `release/`: 빌드 아티팩트 (Git ignore 대상)

------------------------------------------------------------------------

## 🛠️ 빌드 & 실행 명령어

-   `./gradlew bootRun` : 애플리케이션 실행
-   `./gradlew clean build` : 빌드 및 패키징 (Jar)
-   `./gradlew test` : 테스트 실행
-   `./gradlew bootJar` : 배포용 Jar 생성
-   `docker-compose up` : DB/MQ/Redis 등 로컬 실행 환경 기동
-   `./gradlew lint` : 코드 스타일 검사 (Checkstyle, Spotless)

------------------------------------------------------------------------

## 💻 코딩 스타일 & 네이밍 컨벤션

-   언어: Java 21
-   인덴트: 스페이스 4칸
-   클래스: PascalCase (`UserService`, `NoteController`)
-   메서드 & 변수: camelCase (`getUserById`, `noteRepository`)
-   상수: UPPER_SNAKE_CASE (`DEFAULT_PAGE_SIZE`)
-   패키지: 소문자 단어 조합 (`com.example.note.domain`)
-   DTO: `*Request`, `*Response` 접미사 사용
-   JPA 엔티티: 단수형 명사 (`User`, `Note`, `Tag`)
-   SQL 매핑: snake_case (`created_at`, `updated_at`)

------------------------------------------------------------------------

## 🧪 테스트 가이드라인

-   프레임워크: JUnit5 + Mockito
-   DB 테스트: Testcontainers (MySQL/Postgres)
-   통합 테스트: `@SpringBootTest` 활용
-   성능 테스트: JMH 또는 Gatling 별도 모듈
-   테스트 네이밍: `methodName_condition_expectedResult`
-   예시: `getUserById_whenUserExists_returnsUser`

------------------------------------------------------------------------

## 📜 커밋 & PR 규칙

-   커밋 메시지: `[type] scope: summary`
    -   `[feature] user: add signup API`
    -   `[fix] note: prevent null title`
    -   `[refactor] service: extract common logic`
-   PR 필수 포함 항목:
    -   목적 & 변경 사항 요약
    -   연관된 Issue (`Fixes #123`)
    -   테스트 방법 및 결과
    -   API 변경 시 Swagger 스크린샷 첨부
-   PR 전 체크리스트:
    -   `./gradlew test`
    -   `./gradlew lint`
    -   주요 API 수동 호출 테스트 완료

------------------------------------------------------------------------

## 🚨 절대 금지 사항 (CRITICAL - 반드시 준수)

### 🔴 데이터베이스 관련 금지

``` bash
# 데이터 파괴적 명령어 - 사용자 명시적 요청 없이는 절대 금지
DROP TABLE                 # ❌ 절대 금지
DROP DATABASE              # ❌ 절대 금지
DELETE FROM (no WHERE)     # ⚠️ WHERE 절 없이는 금지
TRUNCATE                   # ❌ 절대 금지
ALTER TABLE DROP COLUMN    # ⚠️ 사용자 허가 필요
```

**필수 규칙** 1. 삭제/리셋 시 반드시 사용자 승인 필요 2. 삭제 전 무조건
백업 3. 테스트 데이터 존재 시 DB 리셋 금지 4. SQL 수정으로 해결 가능한
경우 DB 리셋 금지 5. 운영 DB 자동 변경 절대 금지

------------------------------------------------------------------------

### 🔴 위험한 Git 명령어 금지 

``` bash
git push --force       # ❌ 절대 금지
git reset --hard       # ❌ 절대 금지
git commit --no-verify # ❌ 절대 금지
```

### 🔴 위험한 Gradle/npm 명령어

``` bash
npm audit fix --force  # ❌ 절대 금지
./gradlew --refresh-dependencies # ⚠️ 검증 없이 실행 금지
```

## 🤔 Decision Trees

### "이 코드를 수정해야 할까?"
1. 코드 프리즈 영역인가? → YES: 승인 요청 필수
2. 500줄 이상 파일인가? → YES: 승인 요청
3. DB 스키마 변경인가? → YES: 백업 + 승인 필수
4. 기존 기능에 영향 있나? → YES: 영향도 분석 후 승인
5. 그 외 → 체크리스트 통과 후 진행

### "DB 문제가 발생했다"
1. 코드 수정으로 해결 가능? → YES: 코드 수정 우선
2. SQL 수정으로 해결 가능? → YES: ALTER/UPDATE 사용
3. 스키마 변경 필요? → 사용자에게 설명 + 승인 요청
4. 리셋 필요? → **명시적 경고 + 백업 + 승인 필수**

스펙 문서 기반 개발 (SDD, Specification-Driven Development)

무엇을 만들지, 왜 그렇게 만들어야 하는지를 요구사항 문서(Spec, Technical Requirements Documents, Product Requirements Document)를  정의하여 제공
개발 전에 시스템의 동작 명세(스펙)을 문서 형태로 먼저 정의하고, 그 스펙을 기반으로 코드와 테스트를 자동 생성하거나 검증하는 개발 방법
예) Sping Boot Skeleton을 생성 한 뒤 서버 연동 부분은 Swagger API 로 정의 하고 그에 맞춰 Front와 API를 개발하는 방식  혹은 GraphQL을 통해 스키마와 API를 정의 한 뒤 코드를 생헌 하는 방식 

일반 적인 개발 구조 예시 

  • docs/
    • AGENTS.mdc
    • ARCHITECTURE.md
    • API.md
    • DEV/BUILD_RELEASE.mdc
    • STYLE/COLOR.mdc
    • STYLE/DESIGN.mdc
    • PRD/FEAT-ProjectCard.mdc
    • TRD/FEAT-ProjectCard.mdc

각각의 스펙들을 문서로서 제공하고 Agnet로 하여금 스펙 구현에 초점을 두도록 한다. 

스펙 기반 개발에서는 이 문서들 이외에도 요구사항 정의, 구현 계획, 작업 목록 생성, 계획에 따른 구현의 단계를 통해 고급 AI 모델에 대한 높은 의존성을 설정하여 여러 단계로 세분화 하여 개발을 진행한다.


AGENTS.md 에서 참고해야 할 각 문서를 @include 하여 사용할 수 있다. 또한 대규모 모노 레포 구조에서는 각 하위 프로젝트 단위로 AGENTS.md 를 각각 정의하여 지침을 제공할 수 있다.

단, 초기 스펙 정의에 많은 시간이 소모되며 스펙 정의가 명확해야 함 


일반적엔 Agentic 개발 사이클 

필수 문서 가이드 제공 

개발 영역에 적용되어야 할 Base-Rule과 Convention, Guide를 제공하여 개발 우선순위와 코드 탐색, 효율적인 토큰 활용을 도모한다. 

- 각 feature 별 guide 문서를 제공하므로서 (html, md, json 등) 가이드에 따라 구현이 벗어나지 않도록 처리한다. 

- 코딩 패턴, 컨벤션, 팀 작업을 위한 가이드 등 개발 전반에 걸친 내용을 정리하여 제공하면 보다 더 효율적인 문서 기반 개발을 진행하 할수 있다. 


개발과 수정 사이클 

개발 정의 → 질의 → 코드 검토 → 피드백 → 수정 반영 → 검토 → 개선 → 반영 → 승인 → 배포의 사이클을 거치며 개발과 수정을 통해 기능 완성과 품질을 확보

디자인과 UI 영역에서 활용해야 할 요소 고려 

AI로 UI를 구현하는 경우 만족도가 비교적 떨어지고 AI가 개발한걸 알 수 있게끔 결과를 보여주기 때문에(그라디언트, 비슷한 색상) 프로토 타입 수준에서 벗어나기 어렵다. 따라서 완성도 있는 UI를 구성하는게 중요하다.

현재까지 가장 잘 작동하는 방식은 대상 사이트를 추출해서 분석한 뒤 적절한 프롬프트를 사용해서 유사한 결과물을 만들어내거나, 피그마 등을 이용한 디자인 툴을 사용하여 MCP의 도움을 받는 방식이다. 

- 이미지 캡쳐 첨부 

- HTML, CSS 등을 첨부 

- 컬러 추출 및 UI 와이어프레임 제작 후 연동 


그러나 점점 더 좋은 AI Agent들이 나오고 있는 상황이므로, 본인에게 맞는 적절한 툴을 찾는것이 중요하다 다음은 참고할만한 디자인 관련 자료를 정리하였다.

# AGENTS.md – UIXBase (Universal UI Component Library)

## 🧭 Project Overview
**Project Name:** UIXBase  
**Goal:** Build a web & mobile compatible, responsive UI component sharing platform. Users can preview live demos, copy source code (HTML/CSS/JS), and reuse it directly. CSS or theme replacement should instantly restyle components without breaking layout.

**Core Concept:**
> A unified design system platform providing components, templates, and plugins that adapt visually through CSS themes, inspired by Tailwind, Shadcn, and Material.

---

## 🧩 Product Requirements (PRD)

### 1. Product Vision
"Build once, preview everywhere — share, remix, and apply."

- Provide rich visual previews of components in various design styles.
- Support multiple CSS frameworks: Tailwind, Shadcn, Material, Bootstrap.
- Allow full theming flexibility by swapping CSS or variables.
- Extend beyond components — include templates and plugin patterns.

### 2. Core Features
| Feature | Description |
|----------|-------------|
| Component Preview | Live-rendered component demos for browser preview |
| Code Copy | Copy HTML / JS / CSS / React / Vue snippets |
| Style Switcher | Switch between Tailwind, Vanilla CSS, Material themes |
| Responsive Preview | Desktop / tablet / mobile preview tabs |
| Category System | Buttons, Forms, Layouts, Modals, Charts, Icons, etc. |
| Template & Plugin Support | Layout Templates, UI Plugins as reusable modules |
| Theme Override | CSS variable–based theme system, no recompile required |
| Docs & Playground | MDX-based usage docs and editable sandbox |
| AI Assistance | Suggests related components and cross-framework equivalents |
| Export & Embed | iframe embed or zip export for projects |
| UI Layout Structure | UI should adopt a dual-pane layout (similar to GitHub Docs): Left pane features a navigational tree menu; Right pane displays component preview, live code/sandbox, and copy features. |

### 3. Target Users
- Frontend Developers (React, Vue, Next.js, etc.)  
- UI/UX Designers needing design–dev handoff consistency  
- AI Agents referencing frontend structures for code generation  
- Developers seeking ready-to-use responsive UI kits

### 4. User Flow
1. User selects a component category (via the left tree menu).  
2. Preview component with responsive tabs (in the right pane).  
3. Switch theme (Tailwind / Material / Vanilla CSS)  
4. Copy code → paste into HTML/CSS/JS environment  
5. Export template if needed (zip/CDN/embed)

### 5. Product Differentiation
- Framework-agnostic structure  
- CSS-swap-only theming architecture  
- Richer stylistic coverage than Tailwind UI / Shadcn  
- Fully responsive by default (flex + grid + Tailwind breakpoints)

### 6. Success Metrics
| Metric | Target |
|---------|---------|
| Monthly Active Users | ≥ 10,000 |
| Avg. Session Time | ≥ 3 min |
| Code Copy Rate | ≥ 30% |
| New Components / Month | ≥ 50 |
| CSS Swap Error Rate | ≤ 2% |

## ⚙️ Technical Requirements (TRD)

### 1. Architecture Overview
```
Frontend: Next.js + Tailwind + Shadcn + MDX (for documentation)
Backend: Supabase (auth & storage) or Firebase
API Layer: REST + GraphQL hybrid
Build: Vite / Turbopack
Rendering: SSR + CSR hybrid (React Server Components)
UI Layout Specification: Left Pane: Nested Tree Menu (Category, Component List, Templates), Right Pane: Content Area (Live Preview, Code Blocks, Controls)
Each UI element must variously display input forms, HTML elements, and CSS properties, and must mandatorily include @UI.md @COLOR.json @DESIGN.md
```

### 2. Component Structure
```
component/
 ├── core/ (logic)
 ├── style/ (CSS, Tailwind, theme vars)
 └── preview/ (demo UI + code blocks)
```

- Use flex/grid for responsive layout
- All styling controlled by CSS variables (`--color-primary`, `--font-base`)
- Props managed via JSON Schema
- Monaco Editor for live code preview

### 3. Tech Stack
| Layer | Tech |
|-------|------|
| UI Framework | Next.js 15, React 19 |
| Styling | TailwindCSS + Shadcn + CSS Variables |
| Docs | MDX + Contentlayer |
| State | Zustand / Jotai |
| Code Preview | Monaco Editor |
| Build | Turborepo / Vite |
| Auth & Storage | Supabase / Clerk |
| Deployment | Vercel (frontend), Cloudflare R2 (assets) |

### 4. Responsive Design
- Supports desktop, tablet, and mobile breakpoints:
```css
@media (max-width: 640px) { /* mobile */ }
@media (min-width: 641px) and (max-width: 1024px) { /* tablet */ }
@media (min-width: 1025px) { /* desktop */ }
```
- PWA-enabled (offline mode, installable)

Related Files @DESIGN.md 

### 5. Theming System
```css
:root {
  --color-primary: #2563eb;
  --color-bg: #fff;
  --radius-md: 8px;
}
[data-theme="dark"] {
  --color-bg: #0f172a;
}
```
- Swappable color variables and font scales
- Compatible with Tailwind’s theme extension

Related Files @COLOR.json 


### 6. Template & Plugin Framework
- **Template**: layout collections (Dashboard, Blog, etc.)  
- **Plugin**: logic modules (Toast, Modal, DatePicker)

```json
  "type": "plugin",
  "name": "toast",
  "frameworks": ["vanilla", "react"],
  "themes": ["light", "dark"]
}
```

## ✅ Summary
UIXBase merges the practicality of **Shadcn + Tailwind** with the live preview experience of **CodePen**, forming a unified, responsive, and theme-flexible UI library hub.  
Utilizing a GitHub Docs-style two-pane layout for optimal navigation and content viewing, it provides template-, plugin-, and component-level flexibility for rapid frontend prototyping and consistent design system management.
